## ⚠️ 重大教训：GBIF 图片加载问题

### 问题描述
搜索页面（search_result）无法显示植物头图，而植物详情页（plant-detail）可以正常显示。

### 根本原因分析

1. **域名未配置** - GBIF API 域名 `https://api.gbif.org` 需要在微信公众平台配置为 request 合法域名
2. **学名为空** - 云函数返回的 `scientificNameLatin`（拉丁学名）可能为空
3. **匹配模式错误** - 使用了 `strict=true` 严格匹配，应该用 `strict=false` 宽松匹配
4. **加载策略错误** - 同步等待图片加载阻塞了页面显示
5. **没有降级方案** - 拉丁学名为空时没有用中文名作为备选

### 完整解决方案

#### 1. 微信公众平台配置（必须）
```
路径：开发管理 → 开发设置 → 服务器域名
添加：https://api.gbif.org 到 request 合法域名
```

#### 2. 云函数确保返回拉丁学名
文件：`cloudfunctions/getCareGuide/index.js`
```javascript
// Prompt 中强调必填
"scientificNameLatin": "拉丁学名原文（如：Epipremnum aureum）【必填！用于搜索植物图片】",

// 注意里强调
注意：
1. 各字段内容不要重复，互补而非重叠
2. 内容要实用具体，不要泛泛而谈
3. 常见问题要给出具体解决方案
4. **scientificNameLatin 必填！** 这是植物的拉丁学名，用于搜索植物图片
```

#### 3. 搜索页面使用正确学名优先级
文件：`pages/search_result/search_result.js`
```javascript
// 学名优先级：传入学名 > 拉丁学名 > 中文名
const scientificNameToUse = providedScientificName || aiResult.scientificNameLatin || aiResult.scientificName;

// 降级方案：如果学名都为空，用搜索关键词
const searchKeyword = scientificNameToUse || keyword;
```

#### 4. GBIF 调用使用宽松匹配
```javascript
// 使用 strict=false 宽松匹配
url: `https://api.gbif.org/v1/species/match?name=${encodeURIComponent(keyword)}&strict=false`
```

#### 5. 异步加载不阻塞页面
```javascript
// 1. 先显示页面（无图或占位符）
this.setData({ loading: false, error: false, plant });

// 2. 异步加载图片
this.loadPlantImageAsync(searchKeyword);

// 3. 图片加载成功后更新
this.setData({ 'plant.imageUrl': imageUrl });
```

#### 6. 添加详细日志便于调试
```javascript
console.log('[search_result] 学名优先级：拉丁学名=', aiResult.scientificNameLatin, '中文名=', aiResult.scientificName);
console.log('[GBIF] 搜索植物图片:', keyword);
console.log('[GBIF] 物种匹配结果:', speciesKey, scientificName);
console.log('[GBIF] 找到', results.length, '个结果');
console.log('[GBIF] 找到图片:', img.identifier);
```

### 调试检查清单

如果图片不显示，按顺序检查：

1. **检查域名配置**
   - 微信公众平台是否配置了 `https://api.gbif.org`
   - project.config.json 中 `urlCheck` 是否为 true（上线必须为 true）

2. **检查学名是否为空**
   - 查看日志 `[search_result] 学名优先级`
   - 如果拉丁学名为空，检查云函数 Prompt 是否强调必填

3. **检查 GBIF 匹配结果**
   - 查看日志 `[GBIF] 物种匹配结果`
   - 如果 `speciesKey` 为空，说明匹配失败，尝试用中文名

4. **检查图片获取**
   - 查看日志 `[GBIF] 找到 X 个结果`
   - 如果结果为 0，说明该植物在 GBIF 没有图片

5. **检查页面更新**
   - 查看日志 `[search_result] 图片加载成功`
   - 检查 `this.setData({ 'plant.imageUrl': imageUrl })` 是否正确

### 核心原则

1. **域名配置优先** - 涉及外部 API 调用，先检查域名配置
2. **数据完整性** - 确保关键字段（如拉丁学名）有值
3. **降级方案** - 主方案失败时要有备选方案
4. **异步加载** - 不影响用户体验的前提下加载非关键资源
5. **详细日志** - 便于快速定位问题
6. **一次性解决** - 修改前先充分分析，修改后自己验证

### 复用参考

plant-detail 页面的 GBIF 调用是成功的，可以直接复用其实现：
- 使用 `strict=false` 宽松匹配
- 使用回调方式（Promise 封装）
- 先显示占位符，异步加载图片

## 📋 工作方法教训

### 问题：反复修改同一问题

**错误做法：**
1. 没有系统性分析就修改
2. 修改后不自己测试验证
3. 让用户承担测试成本
4. 每次只修复一个点，不考虑其他相关因素

**正确做法：**
1. **充分分析** - 对比成功案例（plant-detail），找出所有差异点
2. **列出清单** - 所有可能的问题点（域名、学名、匹配模式、加载策略、错误处理）
3. **一次性修复** - 同时修复所有问题点
4. **自己验证** - 修改后先自己测试，确认解决再提交
5. **详细日志** - 添加足够的日志便于后续调试

### 核心原则

1. **Text > Brain** - 把分析过程写下来，不要靠脑子记
2. **一次性解决** - 让用户只测试一次，而不是反复编译查看
3. **自己先测** - 提交前自己验证，不要让用户当测试员
4. **系统性思考** - 考虑所有相关因素，不要头痛医头
