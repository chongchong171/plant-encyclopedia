## 一、UI/UX 设计经验

### 1.1 加载动画统一性
**问题：** 多个页面加载动画不一致（有的用 wx.showLoading 灰色弹窗，有的用自定义动画）

**经验：**
- ✅ **统一使用自定义放大镜动画**（绿色镜片 + 扫描光线）
- ❌ **避免使用 wx.showLoading**（灰色半透明卡片遮挡内容）
- 📌 **加载文案统一**："AI 正在识别植物..." / "AI 正在查询植物信息..."

**代码位置：**
- `pages/result_swiper/result_swiper.wxss` - 放大镜动画样式
- `pages/search_result/search_result.wxss` - 搜索结果页加载样式

**复用建议：**
> 任何需要加载动画的场景，优先使用自定义动画，不要调用 wx.showLoading

---

### 1.2 文字内容清理
**问题：** AI 返回的 markdown 格式（`**` 加粗标记）直接显示在界面上，看着乱

**经验：**
- ✅ **前端必须清理 markdown 标记**：`text.replace(/\*\*/g, '')`
- ✅ **统一清理函数**：`cleanMarkdown(text)` 封装到工具函数
- ❌ **不要依赖 AI 返回纯文本**（AI 习惯用 markdown 格式）

**代码位置：**
- `pages/home/home.js` - 识别结果和 AI 对话清理
- `pages/search_result/search_result.js` - 搜索结果清理

**复用建议：**
> 任何 AI 生成的文字内容，显示前都要清理 markdown 标记（`**`, `##`, `###` 等）

---

### 1.3 搜索功能设计
**问题：** 搜索框功能定位混乱（文字搜索 vs 拍照识别）

**经验：**
- ✅ **搜索 = 文字输入**（用户打字查询植物）
- ✅ **识别 = 拍照/相册**（用户上传植物图片）
- ❌ **不要混为一谈**（搜索框不要集成拍照功能）

**页面分工：**
- `pages/discover/discover` - 文字搜索入口
- `pages/search_page/search_page` - 搜索建议/历史记录
- `pages/search_result/search_result` - 搜索结果展示
- `pages/camera/camera` - 拍照识别
- `pages/result_swiper/result_swiper` - 识别结果展示

**复用建议：**
> 搜索和识别是两个独立功能，不要试图合并到一个入口

## 二、页面跳转与返回逻辑

### 2.1 TabBar 页面跳转陷阱
**问题：** 从 TabBar 页面（my-plants）跳转到普通页面（result_swiper），返回时可能回到错误页面

**经验：**
- ✅ **传递 from 参数**：`wx.navigateTo({ url: '...?from=mygarden' })`
- ✅ **返回时判断来源**：根据 from 参数决定 navigateBack 还是 redirectTo
- ❌ **不要依赖 wx.navigateBack()**（TabBar 页面行为特殊）

**代码示例：**
```javascript
// 跳转时传递 from
wx.navigateTo({
  url: `/pages/result_swiper/result_swiper?tmp_filePath=xxx&from=mygarden`
})

// 返回时判断
retryIdentify() {
  const { fromPage } = this.data
  if (fromPage === 'mygarden') {
    wx.redirectTo({ url: '/pages/my-plants/my-plants' })
  } else {
    wx.navigateBack()
  }
}
```

**复用建议：**
> 从 TabBar 页面跳转时，必须传递 from 参数，返回时明确指定跳转路径

---

### 2.2 页面栈管理
**问题：** 多次跳转后页面栈过深，用户返回体验差

**经验：**
- ✅ **使用 redirectTo 替换当前页**（不增加页面栈）
- ✅ **关键流程使用 navigateTo**（允许用户返回）
- ⚠️ **避免连续 3 次以上 navigateTo**

**复用建议：**
> 设计页面流程时，先画跳转图，确保返回路径合理

## 三、数据传递模式

### 3.1 页面间数据传递
**问题：** 识别结果数据如何在页面间传递，避免重复调用 API

**经验：**
- ✅ **完整数据传递**：`identify_data=${encodeURIComponent(JSON.stringify(data))}`
- ✅ **接收页直接使用**：不再调用云函数
- ❌ **不要只传名称**（会导致接收页重新查询）

**代码示例：**
```javascript
// 传递完整数据
wx.navigateTo({
  url: `/pages/search_result/search_result?search_text=xxx&identify_data=${encodeURIComponent(JSON.stringify(res.data))}`
})

// 接收页解析
const identifyData = options.identify_data ? JSON.parse(decodeURIComponent(options.identify_data)) : null
```

**复用建议：**
> 如果 A 页面已经调用 API 获取完整数据，传递给 B 页面时不要只传 ID，要传完整数据对象

---

### 3.2 已知植物优化（秒开）
**问题：** 常见植物（绿萝、多肉等）每次搜索都调用云函数，速度慢

**经验：**
- ✅ **建立本地缓存映射**：植物名称 → 云存储 fileID
- ✅ **已知植物秒开**：直接传递图片 URL，不调用云函数
- ❌ **未知植物才调用云函数**

**代码位置：**
- `pages/discover/discover.js` - IMAGE_MAP 映射表
- `pages/search_result/search_result.js` - getPresetPlant() 预设数据

**复用建议：**
> 对于高频访问的固定数据，建立本地缓存映射，避免重复调用 API

## 四、微信小程序原生 API 使用

### 4.1 wx.showLoading 的坑
**问题：** wx.showLoading 显示灰色半透明卡片，遮挡自定义加载动画

**经验：**
- ✅ **优先使用自定义加载动画**（WXML + WXSS 实现）
- ❌ **避免使用 wx.showLoading**（样式不可控）
- ⚠️ **如果必须使用，确保 wx.hideLoading() 成对出现**

**替代方案：**
```xml
<!-- 自定义加载动画 -->
<view class="loading-container" wx:if="{{loading}}">
  <view class="magnifier-loader">...</view>
  <text class="loading-text">AI 正在查询...</text>
</view>
```

**复用建议：**
> 任何需要加载动画的场景，优先用 WXML+WXSS 实现自定义动画，不要调用 wx.showLoading

---

### 4.2 图片上传与显示
**问题：** 云存储图片 URL 有临时和永久两种，混用会导致图片失效

**经验：**
- ✅ **云存储 fileID**：`cloud://...`（永久有效，小程序直接显示）
- ✅ **临时 URL**：`https://...`（有有效期，需要定期刷新）
- ❌ **不要混用**（统一使用 fileID）

**代码位置：**
- `TOOLS.md` - 云存储配置和 fileID 列表
- `pages/my-plants/my-plants.js` - 图片上传和显示逻辑

**复用建议：**
> 小程序图片存储统一使用云存储 fileID，不要使用临时 URL

## 五、AI 集成经验

### 5.1 AI 回复处理
**问题：** AI 返回内容包含 markdown 格式、模板占位符等

**经验：**
- ✅ **清理 markdown 标记**：`**`, `##`, `###`, `[]` 等
- ✅ **检查模板占位符**：`[xxx]` 需要替换或移除
- ✅ **控制回复长度**：AI 容易输出过长内容

**清理函数：**
```javascript
const cleanMarkdown = (text) => text ? text.replace(/\*\*/g, '').replace(/##+/g, '') : ''
const cleanText = (text) => text ? cleanMarkdown(text).replace(/[\r\n]+/g, ' ').trim() : ''
```

**复用建议：**
> 任何 AI 生成的内容，显示前都要经过清理函数处理

---

### 5.2 AI 工具调用
**问题：** AI 工具调用失败或返回格式错误

**经验：**
- ✅ **云函数封装工具调用**（前端不直接接触 API）
- ✅ **设置超时和重试**（30 秒超时，最多重试 2 次）
- ✅ **降级方案**（API 失败时使用预设数据）

**代码位置：**
- `cloudfunctions/intentClassify/index.js` - AI 意图识别和工具调用
- `pages/home/home.js` - AI 对话处理

**复用建议：**
> AI 工具调用必须有超时、重试、降级三重保障

## 六、云开发最佳实践

### 6.1 云函数性能优化
**问题：** 云函数响应慢（25 秒 → 优化到 8-12 秒）

**经验：**
- ✅ **并行调用多个 API**（PlantNet + 百度 AI + GLM）
- ✅ **设置合理超时**（前端 22 秒，云函数 25 秒）
- ✅ **使用 Promise.race**（防止单个 API 卡住）

**代码位置：**
- `cloudfunctions/identifyPlant/index.js` - 植物识别云函数

**复用建议：**
> 多个外部 API 调用时，使用 Promise.race 设置超时，避免单个 API 卡住整体

---

### 6.2 云存储额度管理
**问题：** 云存储有免费额度（5GB），但需要监控使用情况

**经验：**
- ✅ **图片压缩后再上传**（目标：80KB/张）
- ✅ **使用缩略图策略**（列表页 20KB，详情页 200KB）
- ✅ **删除植物时同步删除图片**（避免垃圾数据）

**待实现：**
- [ ] 图片压缩功能
- [ ] 缩略图自动生成
- [ ] 清理机制（删除植物时同步删图片）

**复用建议：**
> 云存储不是无限的，从第一天就要考虑压缩、缩略图、清理机制

## 八、零记忆 Agent 快速上手指南

如果一个新的 Agent 接手 AI 植物管家项目，应该：

### 8.1 第一步：阅读核心文档
1. `TOOLS.md` - 了解 API 配置和云开发环境
2. `MEMORY.md` - 了解项目历史和重要决策
3. `lessons/README.md` - 了解所有教训（避免重犯）

### 8.2 第二步：理解页面结构
```
pages/
├── home/              - 首页（AI 对话 + 识别入口）
├── discover/          - 发现页（文字搜索）
├── search_page/       - 搜索建议页
├── search_result/     - 搜索结果页
├── camera/            - 拍照识别
├── result_swiper/     - 识别结果页
├── my-plants/         - 我的花园（TabBar）
└── plant-detail/      - 植物详情
```

### 8.3 第三步：理解数据流
```
用户拍照 → camera → result_swiper (识别) → 添加到花园
用户搜索 → discover → search_result (查询) → 查看详情
```

### 8.4 第四步：遵守核心原则
1. **费用红线** - 所有费用操作必须提前确认
2. **API Key 不硬编码** - 全部放在云函数环境变量
3. **修改前看旧版本** - 不要破坏已有的功能
4. **统一加载动画** - 使用自定义放大镜，不用 wx.showLoading
5. **清理 AI 内容** - 显示前清理 markdown 标记

## 九、核心设计原则（可复用到其他项目）

### 9.1 用户体验优先
- 加载动画要美观（自定义 > 原生）
- 文字内容要干净（清理 markdown）
- 返回路径要合理（明确 from 参数）

### 9.2 性能优化
- 已知数据本地缓存（秒开）
- 未知数据异步加载（不阻塞 UI）
- 图片压缩再上传（节省流量）

### 9.3 容错设计
- API 调用有超时
- 失败有降级方案
- 错误提示用户友好

### 9.4 可维护性
- 代码注释清晰
- 文档及时更新
- 教训必须记录
